そのまま読める「話す原稿＋デモ手順」。所要2〜3分。

## 0. ひとこと（つかみ）
「**前回はアプリの作り方の基本を学びました。今回はそのアプリに“AIを組み込む”ための第二章。ChatGPTクローンを1〜14章ぜんぶ作りました。**」

## 背景（なぜやったか）
自分のプロダクトにAIを組み込みたい、という人は周りにも多い。でも「APIをつなぐだけ」では足りないと分かった。
GPTのAPIをどう起動して接続するか、いくらかかるか、使いすぎをどう止めるか、どう記憶させるか——ここを学んだ上で実装するのが大事だと実感した回。

## 1. 作ったもの（TypeScript + Express + OpenAI + Supabase）
- 自分のサーバー経由でLLMと会話（**APIキーはサーバーに隠す。ブラウザに出さない**）
- 会話が続く（LLMは毎回忘れる＝ステートレス。履歴を毎回送り直して継続を作る）
- Googleログイン＋会話をSupabaseに保存（**RLSで自分の会話だけ**。前回の続き）
- **使いすぎ防止**（サーバー側で1日の上限、超過は429で止める＝お金がかからない）
- **トークンあふれ対策**（古い会話をLLMに要約させ、送るのは要約＋直近だけ。全文はDBに残す）
- **メモリ**（会話をまたいでユーザーの事実＝名前・好みを覚える）
- **ストリーミング**（返事が1文字ずつ流れる。SSE）
- **ツール**（電卓を持たせ、大きな掛け算を正確に。function calling）
- server.ts と public/index.html は一度白紙から自分の手でタイプした

## 2. デモ手順（画面を見せながら／今回はローカル localhost:3000）
1. ログイン → 何か質問 → **返事が1文字ずつ流れる**（ストリーミング）
2. 「私の名前はしゅう、辛いもの好き」と伝える → **「新しい会話」ボタン** → まっさらな会話で「名前と好みは？」→ **覚えている**（メモリ＝会話をまたぐ）
3. 上の表示に注目：会話が伸びても **LLMに送るサイズは小さいまま**（要約が効いている）
4. 「12345×6789は？」→ **[道具 calculate を実行]** と出て正確な答え（ツール）
5. （裏の見せ場）ページのソースを表示しても **APIキー(sk-)は出てこない**（鍵を守る）

## 3. 学び（背骨は2つ）
- **背骨①「LLMは毎回忘れる」**。会話が続いて見えるのは、こちらが履歴（要約＋直近）を毎回送り直しているから。記憶を作るのはLLMではなく開発者。
- **背骨②「サーバーで鍵と財布を守る」**。APIキーはブラウザに出さない。使いすぎはサーバー側で止める（フロントの制御は回避される）。
- **APIをつなぐだけでは足りない**と体で分かった。単純につないだだけでは、正確な計算もできないし、会話上に実在のURLも出せない（→ ツールが要る）。
- **金額・使わせない工夫・記憶のさせ方**まで含めて学んでから実装するのが大事。
- これらは Taste Glass（Gemini Live）や、いまのAIサービスの標準構造とそのまま地続き。今回の要約・メモリは Taste Glass の味覚プロファイルに直接使える。

## 4. つまづきポイント（実体験の共有）
- **APIキーをターミナルに貼っても動かない**。→ 鍵はエディタで `.env` に書く（ターミナルはコマンドを打つ場所。貼ると履歴に鍵が残る危険も）。
- **OpenAIは前払い式**。カード登録だけではダメで、クレジットを少額チャージしないと quota エラー。Auto recharge はオフにして青天井を防ぐ。
- **Supabase無料は1アカウント2プロジェクトまで**。3つ目が作れず、Twitterクローンのプロジェクトに相乗り（表を足すだけで済んだ）。
- **Googleログインの戻り先**。`localhost:3000` を Supabase の Redirect URLs に追加する必要があった。
- **HTMLの貼る位置**。動くコードは `<script>` の中、部品の読み込み(CDN)は使う前に置く。外や後ろに出ると無視されて動かない。
- **ブラウザ用(CDN)とサーバー用(npm)は別物**。server.ts で使う部品は `npm install` が要る（忘れて ERR_MODULE_NOT_FOUND）。
- **ストリーミングで改行が消える**。断片をそのまま流すと多行の返事が崩れる → 断片をJSON化して送って解決。
- **外部サービスは落ちる**。作業中に Supabase の一時障害の表示が出た。第14章「頼っている外部が止まる」がリアルに分かった。

## 5. AI活用の進め方（前回の型を継続）
- **1章ずつ**進める（動くもの＋「なぜ」を自分の言葉で言える状態）。一気に作らせない。
- 途中でAIが先回りして全部書いてしまった場面あり → コードを答え合わせ用に避け、**白紙から自分で打ち直して**取り戻した。「自分の手で作る」を守るのが学びには効く。
- 教材どおりに再現（勝手にアレンジしない）。困りどころは正直に共有。

## 補足：本命は Gemini Live。今日の学びはどう効くか
- Gemini Live が最初から持つ：ストリーミング（音声）／ツールの仕組み／**1セッション中の記憶**（RESTのように毎回全送りしなくてよい）。
- 今日と同じで自分で作る：**会話をまたぐ記憶＝味覚プロファイル（第10章）**（Liveはセッションが切れると忘れる）／**鍵を守る（第2章、一時トークン）**／**使いすぎ・お金（第6章、音声は高いので尚更）**。
- つまり「なぜ記憶・鍵・上限が要るか」の理解はGPTでもGeminiでも同じ。今日作った memory / 鍵 / 上限が、Taste Glass の設計図になる。
